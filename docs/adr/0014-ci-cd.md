# ADR-0014 — Path-scoped monorepo CI/CD with executable boundary rules

**Status:** Accepted (pending approval) · **Date:** 2026-07-04

## Context
A monorepo spanning iOS, backend, and dashboard needs fast CI, safe delivery, and — crucially — a way to keep the architecture's portability/boundary guarantees from eroding over time. Rules that live only in docs decay.

## Decision
**Path-scoped CI**: each PR runs only the jobs for changed areas (iOS/backend/dashboard/contract/config). Mandatory gates: lint/format, **boundary lints** (no `import Supabase` outside `Networking`; no raw color/font literals outside `DesignSystem`; no Feature→Feature imports), **contract validation** (OpenAPI + JSON Schema + configs), unit/snapshot/RLS/contract tests, and a DI graph-resolution test. Delivery: backend migrations + Edge Functions to staging → manual approval → production; iOS via Fastlane lanes per client/env (TestFlight → App Store) parameterized by the white-label pipeline; dashboard via standard web deploy. Semantic versioning; `CHANGELOG` maintained; migrations forward-only and reproducible from the repo.

## Alternatives considered
- **Single monolithic CI job for the whole repo:** simplest config, but slow and wasteful; rejected in favor of path scoping.
- **Boundary rules enforced only by review/docs:** rejected — guarantees must be executable or they rot (this is the mechanism that makes "replace Supabase later" credible).
- **Auto-deploy to production on merge:** rejected for v1 — a manual approval gate for backend/prod is prudent while the platform stabilizes.

## Consequences
- (+) Fast CI; architecture rules enforced continuously; reproducible deploys.
- (+) White-label builds are first-class CI citizens.
- (−) More CI tooling to author/maintain (path filters, custom lints). Accepted — it protects the core theses.
