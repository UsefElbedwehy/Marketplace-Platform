# iOS Architecture

> Living document. Full architecture: [planning/02-ios-architecture.md](planning/02-ios-architecture.md).

Clean Architecture + MVVM-C, Coordinator pattern, Factory DI, SPM modularization, SwiftData as offline cache. See ADRs [0001](adr/0001-clean-architecture-ios.md), [0012](adr/0012-dependency-injection.md), [0013](adr/0013-local-persistence.md), [0015](adr/0015-ios-modularization.md).

**Status:** Phase 3 (foundation) and the Phase 4 golden path are both **done** and verified live in the iOS Simulator. Nine SPM packages (`Core`, `DomainKit`, `Networking`, `Configuration`, `DesignSystem`, `DataKit`, `DynamicForms`, `Features/Listings`, `Features/Search`) plus the `App` composition root — each real Swift, unit-tested (112 logic tests across the seven non-snapshot packages; `DesignSystem` additionally has 9 logic + 3 snapshot tests, `DynamicForms` 13 logic + 5 snapshot tests, both run against a real iOS Simulator via `xcodebuild`). Factory DI (ADR-0012), Clean Architecture layering (ADR-0001, enforced by SPM target dependency declarations — an illegal import is a compile error), MVVM-C with a top-level `AppCoordinator` + `TabCoordinator` (`ios/App/Sources/App/Coordinators/`), SwiftData as an offline cache (ADR-0013, `Configuration`'s `CachedConfigRecord`/`CachedThemeRecord`).

The App target itself is wrapped into a real, installable `.app` by [Tuist](https://tuist.dev) (`ios/Tuist.swift` + `ios/Tuist/Package.swift` + `ios/Project.swift`) — a bare SPM `.iOSApplication` product builds via `xcodebuild` but doesn't get packaged into a `.app` bundle by the CLI alone. See [ios/App/README.md](../ios/App/README.md) for the build/run/test commands and what's been verified (boot → auth → themed tab shell, in light/dark/RTL, via real `XCUITest`s since `xcrun simctl` has no touch-input primitive).

Real GoTrue-backed auth (email/OTP/Apple/Google, ADR-0007) is unimplemented backend work — the App's sign-in screen is a dev-only identity picker mirroring the dashboard's `DevIdentitySwitcher`, against the same `/v1/dev-auth` endpoint.

## DynamicForms + Features (Phase 4 ⭐)

`DynamicForms` is the on-device half of the [Dynamic Category & Attribute Engine](planning/05-dynamic-schema-engine.md): a `FieldRendererRegistry` mapping every `AttributeInputType` to a SwiftUI renderer, plus `DynamicFormState` (dependency resolution — `visible_when`/`options_filtered_by` — ported from the dashboard's `DynamicFieldPreview.tsx`). `Features/Listings` (create/feed/detail/my-listings) and `Features/Search` (dynamic filters + results) consume it; neither Feature package imports the other (`tooling/lint/boundaries.sh`-enforced) — a small presentational view (the listing row) is deliberately duplicated between them rather than shared, since no lower layer can host a `Listing`-typed view without crossing the Domain/DesignSystem boundary.

**Cross-module DI registration** (the pattern Phase 4 established, used for every repository/use-case a Feature package needs): the `Container` extension *declaration* — a computed property returning `Factory<SomeProtocol>` with a `fatalError` placeholder body — lives in `DomainKit`, the lowest layer every Feature can see. The App's composition root supplies the concrete implementation later via Factory's `.register(factory:)` override (`container.someProperty.register { ConcreteImpl(...) }`). This works because a `FactoryKey`'s identity is the accessor's return type plus its property name, not the file/line it was declared at — so a declaration in `DomainKit` and an override from `App` target the same registration. ViewModels take the resolved value via a constructor default (`init(useCase: SomeUseCase = Container.shared.someUseCase())`) rather than a bare `@Injected` property wrapper, so tests can inject a fake directly with zero global-container mutation.

Verified end-to-end (`ios/App/UITests/GoldenPathUITests.swift`): created a Car, an Apartment, and a Phone — three structurally different schemas (option fields + a dependency; pure number/bool fields with no options at all; the dependency pattern again in a different vertical) — through the identical `CreateListingView` code, walked one through the real moderation pipeline to `published`, and confirmed Search's dynamic filter surfaced it. `SchemaLiveEditUITests.swift` separately confirms a server-side schema edit (through the real `catalog_editor` write API) appears in an already-open create-listing form after a plain re-fetch, no rebuild. See [ROADMAP.md](ROADMAP.md#phase-4--ios-golden-path-dynamicforms--listings-) and [CHANGELOG.md](../CHANGELOG.md) for what shipped, including two real bugs the UI test caught (an accessibility-collapsing container and a `@State` initial-value race).
