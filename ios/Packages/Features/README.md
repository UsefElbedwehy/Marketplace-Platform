# Features

One SPM package per feature module (Authentication, Listings, Search, Favorites, Chat, Notifications, Profile, Seller Dashboard, Subscriptions, Payments, Wallet, Reports, Moderation, CMS, Analytics, Settings, Support, …). **A Features/X package must never import another Features/Y package** — enforced by `tooling/lint/boundaries.sh`. Cross-feature interaction goes through `DomainKit` use cases/protocols or coordinator route enums. Each feature package ships its own `README.md` / `Architecture.md` / `Flow.md` / `API.md` / `Testing.md` / `Future.md` per the repository documentation standard. See [docs/planning/01-system-architecture.md §4](../../../docs/planning/01-system-architecture.md#4-module-boundaries--dependency-rules).

**Status:** `Listings` and `Search` are implemented (Phase 4 golden path) — see `Features/Listings/README.md` and `Features/Search/README.md`. Neither imports the other; a small presentational view (the listing row) is deliberately duplicated between them rather than shared, since no lower layer (`DesignSystem` can't depend on `DomainKit`; `DomainKit` can't depend on `DesignSystem`) can host a `Listing`-typed view. Authentication has no dedicated Feature package — the dev-only sign-in flow lives directly in `App` (Phase 1's real GoTrue-backed auth is unimplemented backend work, ADR-0007). Remaining Feature modules (Favorites, Chat, Notifications, …) are Phase 6+.
