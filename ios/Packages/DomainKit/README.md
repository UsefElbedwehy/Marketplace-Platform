# DomainKit

Pure Swift domain layer: entities, use case protocols, repository protocols. No imports beyond Foundation — this is what stays portable no matter what backend or UI framework sits around it. See [docs/planning/02-ios-architecture.md §1](../../../docs/planning/02-ios-architecture.md#1-layered-architecture) and [ADR-0001](../../../docs/adr/0001-clean-architecture-ios.md).

**Status:** implemented and tested (`swift test` — 16/16 passing). Depends on Core only. Ships the boot-time slice: `AppConfiguration` entity (locales/currencies/modules/featureFlags/support — a trimmed domain projection of `Configuration`'s full DTO), `AuthSession`/`DevIdentity` entities, `ConfigRepository`/`AuthRepository` protocols, and `FetchBootstrapConfigUseCase`/`AuthUseCase`. Deliberately has **no** theme entity/repository — theme has no business logic and lives entirely in the `Configuration` → `DesignSystem` lane (see `DesignSystem/Theme/ThemeStore.swift`'s doc comment).

**Extended in Phase 4** with the golden path's entities: `CatalogSchema.swift` (`CategoryTreeNode`, `ComposedSchema`/`SchemaGroup`/`SchemaField`, `AttributeDependency`/`DependencyRule`, `AttributeOption`) and `Listing.swift` (`Listing`, `AttributeValue`, `CreateListingDraft`, `Page<T>`, `ListingFilters` — a `mine: Bool` flag rather than an arbitrary `ownerId`, since the contract's `owner=me` query param only ever resolves to the caller's own id server-side), plus `CategoryRepository`/`ListingRepository` protocols and `CatalogUseCase`/`ListingUseCase`.

## Cross-module DI registration points (`DIRegistrationPoints.swift`)

Every Factory `Container` extension a Feature package needs to resolve (`categoryRepository`, `listingRepository`, `catalogUseCase`, `listingUseCase`, plus the pre-existing `configRepository`/`authRepository`/`fetchBootstrapConfigUseCase`/`authUseCase`) is *declared* here — the lowest layer visible from every Feature — as a computed `Factory<Protocol>` with a `fatalError` placeholder body. The App composition root supplies the concrete implementation later via `container.someProperty.register { ConcreteImpl(...) }`. This works because a `FactoryKey`'s identity is the accessor's return type plus its property name, not the file/line it's declared at, so a declaration here and an override from `App` target the same registration. `Core`'s own `appLogger` is the one exception — it has a real default (`OSAppLogger.shared`) declared directly in `Core`, since it never needs an App-specific override.
