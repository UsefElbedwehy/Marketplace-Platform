# Networking

`APIClient` / `APIEndpoint` + interceptors (auth token injection, refresh-on-401, retry/backoff, correlation id, error-envelope decoding). **The only place `import Supabase` is allowed** — enforced by `tooling/lint/boundaries.sh`. This is the literal mechanism behind "the app never depends on Supabase directly." See [docs/planning/08-api-auth.md §4](../../../docs/planning/08-api-auth.md#4-client-networking-layer-ios--the-abstraction-that-protects-portability) and [ADR-0002](../../../docs/adr/0002-backend-bff-edge-functions.md).

**Status:** implemented and tested (`swift test` — 13/13 passing, including real Keychain round-trips and `URLProtocol`-stubbed HTTP tests). Ships `APIEndpoint`, `APIClient`/`URLSessionAPIClient` (decodes the standard `{data:}`/`{error:}` envelope, maps HTTP status + envelope `code` to `DomainError`), a Keychain-backed `TokenStore` actor, and `DevAuthEndpoint` (the local-only `/v1/dev-auth` client, deliberately excluded from `contract/openapi` — mirrors the backend's `v1-dev-auth`). No Supabase SDK dependency at all yet — v1 auth is REST to the Edge Functions gateway, so the "no Supabase import outside Networking" boundary is presently trivial (and CI-enforced going forward via `tooling/lint/boundaries.sh`).

**Extended in Phase 4** with `sendCollection<T>` on `APIClient` + `CollectionEnvelope`/`PaginationMeta` — the contract's list endpoints return `{"data": [...], "page": {"nextCursor": ..., "hasMore": ...}}`, where `page` is a *sibling* of `data`, not nested inside it. Reusing the existing `send<T>` with `T = [Item]` would technically decode without error (JSONDecoder just ignores the unmatched `page` key) but silently discard pagination state — `sendCollection` is a genuinely separate capability, not a workaround.
