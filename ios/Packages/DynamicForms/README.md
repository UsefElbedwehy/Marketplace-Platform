# DynamicForms

The iOS realization of the [Dynamic Category & Attribute Engine](../../../docs/planning/05-dynamic-schema-engine.md): a field-type renderer registry that turns backend attribute metadata into SwiftUI forms and filters — schema-driven, never screen-driven. The same renderer produces the Cars form and the Apartments form from different metadata. See [docs/planning/02-ios-architecture.md §6](../../../docs/planning/02-ios-architecture.md#6-dynamic-form-rendering-the-ios-side-of-the-flagship).

**Status:** implemented and tested — 13 logic tests (`swift test`) + 5 real snapshot tests (`xcodebuild test -scheme DynamicForms -destination 'platform=iOS Simulator,name=...'` — plain `swift test` renders through macOS/AppKit, not iOS, so the snapshot suite is `#if canImport(UIKit)`-guarded). Depends on Core, DomainKit, DesignSystem (+ Configuration, test-only, for the bundled theme fixture). Ships:

- **`FieldRenderer`** — a protocol with one requirement, `body(field:options:value:) -> AnyView`, marked `@MainActor` on the method only (not the whole protocol — putting `@MainActor` on the protocol made every conforming struct's initializer implicitly actor-isolated, which broke `FieldRendererRegistry.standard`'s plain dictionary literal built in a nonisolated static context).
- **`FieldRendererRegistry`** — a `[AttributeInputType: any FieldRenderer]` dictionary. Adding a new field type means adding one entry; `DynamicFieldView`, the single dispatch point, never changes (Open/Closed).
- **Renderers** (`Renderers/`): `TextFieldRenderer`/`TextAreaRenderer`, `NumberRenderer` (one numeric `TextField` covers both `stepper` and `slider` — the seed data only exercises `stepper`), `DropdownRenderer` (`Menu`-based), `ChipsRenderer` (+ a custom `FlowLayout: Layout` for wrapping), `SwitchRenderer`, `DateRenderer` (ISO `yyyy-MM-dd`), and `PlaceholderRenderer` for `media`/`location` — explicitly not implemented, matching the dashboard's own precedent rather than half-building an upload/map control nothing in the seed schema needs yet.
- **`DynamicFormState`** (`@Observable @MainActor`) — owns `values`/`fieldErrors` for a `ComposedSchema`, and the dependency engine ported faithfully from the dashboard's `dashboard/src/components/DynamicFieldPreview.tsx`: `isVisible(_:)` (a field is hidden unless another field's value equals a `visible_when` condition) and `visibleOptions(for:)` (`options_filtered_by` — resolves the parent field's *selected value* to the parent's *option id*, then matches option rows whose `parentOptionId` equals it). `validate()` checks required/min/max/maxLength against `SchemaField`'s validation metadata; `applyServerErrors(_:)` maps a 422's field errors back onto the form; `submittableAttributes` strips values behind a currently-hidden `visible_when` field before a create/update request goes out.
- **`DynamicFieldView`/`DynamicFormView`** — the actual dispatch point and the top-level form renderer (grouped by `SchemaGroup`, in `SchemaField.sortOrder`).

Every renderer's editable control carries `.accessibilityIdentifier("dynamic-field.<key>")`, which is how `ios/App/UITests/GoldenPathUITests.swift` locates and fills each field end-to-end. `Tests/DynamicFormsTests/Fixtures.swift`'s `CarsSchemaFixture` exercises brand→model `options_filtered_by`, a `visible_when` condition, numeric min/max, and text `maxLength` together.

Consumed by `Features/Listings` (`CreateListingView`'s form, driven by a category's live schema) and `Features/Search` (the equality half of the filter sheet — filterable *number* fields get a separate min/max range row in `Features/Search`, outside this renderer's single-value model). See [ROADMAP.md](../../../docs/ROADMAP.md#phase-4--ios-golden-path-dynamicforms--listings-) for the golden-path verification.

## Snapshot tests (`Tests/DynamicFormsTests/DynamicFormSnapshotTests.swift`)

A small curated matrix across two structurally different fixtures (`CarsSchemaFixture` — dropdowns + a Brand→Model dependency; `ApartmentsSchemaFixture` — pure number/bool fields, no options or dependencies at all) × representative slices of {LTR/RTL} × {light/dark}: `carsFormLight`, `carsFormDark`, `carsFormRTL`, `apartmentsFormLight`, `apartmentsFormRTLDark`. Reference images live in `Tests/DynamicFormsTests/__Snapshots__/`; re-record after an intentional visual change with `RECORD_SNAPSHOTS=1 xcodebuild test -scheme DynamicForms -destination 'platform=iOS Simulator,name=iPhone 17'`, then inspect the new PNGs before committing them.

Note: running `xcodebuild test -scheme <package>` directly against a plain SPM package materializes a throwaway `<Package>.xcodeproj` as a build-system side effect (gitignored — `ios/Packages/**/*.xcodeproj/`) — if a scheme mysteriously reports "not currently configured for the test action," delete that stray `.xcodeproj` and retry.
