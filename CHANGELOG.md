### Added — Phase 4: iOS golden path — DynamicForms + Listings ⭐

The thesis proof on-device: *schema-driven, not screen-driven*. Three new SPM packages (`DynamicForms`, `Features/Listings`, `Features/Search`) plus extensions to `DomainKit`/`Networking`/`DataKit`/`Core`, built bottom-up with real `swift test` runs, then wired into the App and verified end-to-end in the Simulator against the real local backend gateway.

- **DomainKit**: `CatalogSchema.swift` (`CategoryTreeNode`, `ComposedSchema`/`SchemaGroup`/`SchemaField`, `AttributeDependency`/`DependencyRule`, `AttributeOption`) and `Listing.swift` (`Listing`, `AttributeValue`, `CreateListingDraft`, `Page<T>`, `ListingFilters` — `mine: Bool` rather than an arbitrary `ownerId`, since the contract's `owner=me` only ever resolves to the caller's own id), `CategoryRepository`/`ListingRepository` protocols, `CatalogUseCase`/`ListingUseCase`. A new `DIRegistrationPoints.swift` moves every cross-module Factory `Container` extension (`categoryRepository`, `listingRepository`, `catalogUseCase`, `listingUseCase`, plus the pre-existing config/auth ones) here — the lowest layer visible from every Feature package — with `fatalError` placeholders; the App composition root supplies the concrete implementation via Factory's `.register(factory:)` override, which works because a `FactoryKey`'s identity is the accessor's type + name, not its file/line. 16 tests (up from 5).
- **Networking**: a new `sendCollection<T>` on `APIClient` + `CollectionEnvelope`/`PaginationMeta` — the contract's `{"data": [...], "page": {...}}` collection shape has `page` as a *sibling* of `data`, not nested inside it, so reusing `send<T>` with `T = [Item]` would silently discard `nextCursor`/`hasMore` rather than actually decode them. 13 tests (up from 11).
- **DataKit**: `CategoryRepositoryImpl` + `ListingRepositoryImpl` (the only places `CatalogDTO`/`ListingDTO`/`JSONScalar` translate to/from `DomainKit` entities), `JSONScalar` (a minimal internal Codable enum for the contract's heterogeneous JSON blobs — `attributesIndex`, `validation`, `condition.equals` — without a full recursive JSON-value type). 18 tests (up from 7).
- **DynamicForms** (new package) — the field-type registry: `FieldRenderer` protocol (`@MainActor` on the `body` method only, not the whole protocol — keeping it at the protocol level made every conforming type's initializer implicitly actor-isolated, breaking the registry's plain dictionary literal) + `FieldRendererRegistry.standard`, one renderer per `AttributeInputType` (text/textarea/number/dropdown/chips/switch/date, plus an explicit "not implemented yet" placeholder for media/map, matching the dashboard's own precedent). `DynamicFormState` ports the dependency-resolution logic faithfully from `dashboard/src/components/DynamicFieldPreview.tsx` (`visible_when`, `options_filtered_by` — filtering a field's options by the parent's *selected value* resolved to the parent's *option id*). 13 tests.
- **Features/Listings** (new package) — `CategoryPickerView` (sheet-presented 2-level drill-down), `CreateListingView` (the golden-path proof: one view renders a full form for any category from its live schema alone), `ListingFeedView`, `ListingDetailView` (schema-projected attribute rows), `MyListingsView` (owner actions + a moderator's approve/reject queue, mirroring the dashboard's `OWNER_ACTIONS`/moderation queue). 18 tests.
- **Features/Search** (new package, does not import Listings — Feature isolation) — dynamic filters built from the same `DynamicForms` registry (equality fields go through `DynamicFormView`; filterable *number* fields get a separate min/max range row, outside that renderer's single-value model), category-scoped results with cursor pagination. `SearchResultRowView` intentionally duplicates `Listings.ListingRowView` — no shared layer can host a `Listing`-typed view without violating the Domain/DesignSystem boundary. 8 tests.
- **App**: `registerDependencies()` supplies the concrete `CategoryRepositoryImpl`/`ListingRepositoryImpl`/`DefaultCatalogUseCase`/`DefaultListingUseCase` via the new DI registration points; Home/Search/Sell tabs now render `ListingFeedView`/`SearchView`/`MyListingsView` instead of Phase 3's placeholders (Chat remains a placeholder — Phase 6).
- **Verified live in the iOS Simulator**, against the real local backend gateway and a real Postgres, via `XCUITest`: created a Car (BMW X5, exercising the Brand→Model dependent-options pattern), an Apartment (bedrooms/bathrooms/area — a schema with *no* option fields or dependencies at all, structurally the opposite of Cars/Phones), and a Phone (Apple/iPhone 15, the same dependent-options pattern in a different vertical) — all three through the identical `CreateListingView`/`DynamicFormView` code. Walked the Car listing through the real moderation pipeline (submit for review → sign in as a moderator → approve) and confirmed Search's dynamic filter (`brand = BMW`) surfaced it — proving browse+filter against a genuinely published listing, not just a client-side stub. Separately verified the "edit a schema server-side → app reflects it after refresh, no rebuild" exit criterion: added a new attribute to the Phones category through the real `catalog_editor`-gated write API while the app was already running, then confirmed the field appeared in the already-open create-listing form after re-selecting the same category — no relaunch, no rebuild.
- **Two real bugs caught by the golden-path UI test, both fixed at the source**:
  1. `MyListingsView`'s per-row `.accessibilityIdentifier` was attached directly to a `VStack` containing multiple buttons (the listing's own tap target plus every owner/moderator action button). SwiftUI collapses a container into a single opaque accessibility element the moment *any* accessibility modifier is attached to it, unless the container also declares `.accessibilityElement(children: .contain)` — so "Submit for review"/"Approve" were unreachable to VoiceOver (and to the UI test) even though they were visibly on screen. Fixed by adding `.accessibilityElement(children: .contain)`.
  2. `SellTabView` computed `isModerator` asynchronously (`.task { session = await authUseCase.currentSession() }`) and passed the `@State` into `MyListingsView(isModerator:)`, which forwarded it straight into `MyListingsViewModel`'s `let isModerator: Bool` via `State(initialValue:)`. `@State`'s initial value is captured once, at the view's first render — by the time the async lookup resolved `true`, the child's state had already locked in `false` from before the session was known, permanently (a moderator could never see the moderation queue, no matter how they signed in). Fixed by having `MyListingsViewModel` resolve its own `isModerator` fresh inside `load()` via an injected `AuthUseCase`, removing the parameter (and the race) entirely.
- **DynamicForms snapshot tests** (roadmap Phase 4's third exit criterion): a curated matrix across two structurally different fixtures (Cars-shaped: dropdowns + a Brand→Model dependency; Apartments-shaped: pure number/bool fields, no options at all) × representative {LTR/RTL} × {light/dark} slices — 5 tests, run via `xcodebuild test -scheme DynamicForms` against a real iOS Simulator (plain `swift test` renders through macOS/AppKit, not iOS). Recorded and re-verified against the recorded baselines in a second run, per this project's snapshot-testing convention. **Phase 4 is now fully done.**
  - Along the way, discovered that `xcodebuild test -scheme <package>` against a bare SPM package directory materializes a throwaway `<Package>.xcodeproj` as a build-system side effect — a stale one (from an earlier Xcode session) caused both `DesignSystem`'s and `DynamicForms`' snapshot schemes to report "not currently configured for the test action" until deleted. Now gitignored (`ios/Packages/**/*.xcodeproj/`) so it can't recur.

### Added — Phase 3: iOS foundation ⭐

Six new SPM packages (`ios/Packages/{Core,DomainKit,Networking,Configuration,DesignSystem,DataKit}`) plus the `App` composition root — built bottom-up, each verified with real `swift test` runs before moving to the next, matching the pattern established for the backend/dashboard work.

- **Core**: `@_exported import Factory` (the DI facade every other module resolves through — ADR-0012), `DomainError`/`FieldError` taxonomy, `AppLogger`/`OSAppLogger`, `AppLocale`. 6 tests.
- **DomainKit**: `AppConfiguration`/`AuthSession`/`DevIdentity` entities, `ConfigRepository`/`AuthRepository` protocols, `FetchBootstrapConfigUseCase`/`AuthUseCase`. No theme entity — theme has no business logic and stays entirely in the `Configuration` → `DesignSystem` lane, never touching Domain/Data. 5 tests.
- **Networking**: `APIEndpoint`/`APIClient`/`URLSessionAPIClient` (envelope decoding, HTTP-status + error-code → `DomainError` mapping), a Keychain-backed `TokenStore` actor, `DevAuthEndpoint` (`/v1/dev-auth` client). Zero Supabase SDK usage yet. 11 tests, including real Keychain round-trips (confirmed headless Keychain access works with no permission prompt in this environment) and `URLProtocol`-stubbed HTTP tests.
- **Configuration**: `ConfigDTO`/`ThemeDTO` (field-for-field mirrors of `configs/clients/<id>/*.json`), a bundled build-time default (copied from `configs/clients/default/*.json`, so the two can't silently drift), a SwiftData cache (ADR-0013), and `ConfigurationDataSource`'s cache-then-network policy. 11 tests.
- **DesignSystem**: `Theme`/`SemanticColors`/`Typography`/`ThemeShape` resolved from `ThemeDTO`, `ThemedRoot` (environment injection + color-scheme-reactive semantic colors + root-level `.tint()`), `ThemeStore`, and a first component slice (buttons, card, badge, text field, loading/shimmer, empty/error/offline states). 9 logic tests + 3 real snapshot tests (`swift-snapshot-testing`, run via `xcodebuild test` against an iOS Simulator — plain `swift test` would render through macOS/AppKit, not iOS).
- **DataKit**: `ConfigRepositoryImpl`, `AuthRepositoryImpl` — the only places `Configuration`'s/`Networking`'s DTOs get translated into `DomainKit` entities. 7 tests.
- **App**: Factory registrations, `AppCoordinator` (boot → auth → main, concurrent config+theme fetch via `async let`) + `TabCoordinator`, a dev-only sign-in screen (mirrors the dashboard's `DevIdentitySwitcher` against `/v1/dev-auth`), a themed 5-tab shell, and `Resources/Localizable.xcstrings` (English + Arabic).
- **Tuist** (`ios/Tuist.swift`, `ios/Tuist/Package.swift`, `ios/Project.swift`) generates the real `.xcodeproj`/`.xcworkspace` the App target needs to become an installable `.app` — a bare SPM `.iOSApplication` product builds via `xcodebuild build` but doesn't get wrapped into a bundle by the CLI alone (verified: only a raw Mach-O comes out). The packages stay plain SPM; Tuist only wraps App.
- **Verified live in the iOS Simulator**, against the real local backend gateway: boot → concurrent config+theme fetch → themed dev sign-in screen → real `/v1/dev-auth` round trip → Keychain-persisted session → themed tab shell → Profile reads the session back → sign-out clears it. Also verified in dark mode (`xcrun simctl ui ... appearance dark`) and in Arabic/RTL (translated strings **and** mirrored layout, via a UI test launched with `-AppleLanguages (ar)`). Driven by two real `XCUITest`s (`ios/App/UITests/`) since `xcrun simctl` has no tap/touch primitive for anything beyond a screenshot.
- Extended `tooling/lint/boundaries.sh` to also scan `ios/App` (previously only `ios/Packages`) for the Supabase-isolation and raw-color/font-literal checks, and re-verified it actually fails on injected violations before confirming it passes clean. Added a `swift test`-based CI job (`ios-unit-tests`, `macos-latest`) for the five non-UI packages.
- **Known cosmetic finding, not chased further**: iOS 26's new floating tab bar didn't pick up `.tint()` for the selected tab's color in manual testing, though the identical color renders correctly everywhere else (confirmed via snapshot tests). A static `UITabBar.appearance()` fix would conflict with the live theme-hot-swap guarantee (ADR-0005), so it's left as a known finding rather than worked around.

### Added — Admin user/role management (Phase 5 slice)

The dashboard's Users screen (one of the CMS-managed resources named in the foundation doc) exposed a real architectural gap: the original `identity.prevent_self_role_escalation` trigger blocked **every** non-service_role change to `app_role`, including an admin trying to promote someone else — there was no way to manage roles except direct service_role DB access.

- **Database**: new forward-only migration `20260704155058_identity_admin_role_management.sql` adds a `profile_update_admin` RLS policy (admin/super_admin may update another tenant member's profile row) and rewrites the trigger: always block a user from changing their own `app_role`/`tenant_id`; allow admin/super_admin to change **another** user's `app_role`; `tenant_id` stays service_role-only regardless of caller. Manually verified via raw psql before formalizing.
- **Backend**: `backend/src/user_service.ts` — `fetchProfiles`, `fetchProfileById`, `updateUserRole` (rejects invalid roles, rejects an actor changing their own role, surfaces RLS's silent-filter-on-UPDATE as a clean 404). 6 unit tests with mocked `QueryExecutor`. `GET /v1/users` + `PATCH /v1/users/{id}` wired into a new `v1-users` Edge Function (admin-only — RLS is the row-access gate, the handler turns a non-admin's empty RLS result into a clean 403), OpenAPI contract extended (`users` tag, `UserProfile` schema), types regenerated.
- **DB test suite**: 7 new RLS assertions in `backend/tests/sql/020_identity_tests.sql` (41 total, up from 34) — admin changes another's role (succeeds and persists), admin cannot change their own role (raises), tenant_id unchangeable even by admin (raises), a non-admin's attempt to change someone else's role runs without error but has no effect (RLS silently filters the UPDATE).
- **Dashboard**: a new `/users` page — admin-only (with a clean message for everyone else), lists tenant members with name/email/role/joined date, a role dropdown per row, the caller's own row disabled with an explanatory tooltip. Verified live: promoted Dev Buyer from seller back to buyer as Dev Admin (persisted, confirmed via a re-fetch), confirmed the admin's own row can't be changed via the UI, and independently confirmed via curl that the server rejects an admin's self-role-change (403) and a non-admin's `/v1/users` request (403) even bypassing the UI entirely.
- **A real test-harness bug caught along the way**: `test.act_as_service()` reset the Postgres role but not the simulated `request.jwt.claims`, so `auth.role()` (which reads that GUC rather than the actual session role) kept reporting the previous test's role — made a legitimate service-role cleanup UPDATE fail the trigger's `auth.role() = 'service_role'` check. Not a production bug: the real request path (`withRequestContext`) always sets role and claims together in the same call. Fixed by having `act_as_service()` set the claims blob too.

### Added — Listings management + moderation (Phase 6/7 slice)

Continuing the reprioritized focus (backend + dashboard first): closes the loop from listing creation through publication.

- **Backend**: `updateListingStatus` in `backend/src/listing_service.ts` — a status-transition state machine (`draft → pending_review → published/rejected`, `published → archived/sold`, etc.) layered on top of RLS (which gates row access; this layer additionally enforces which transitions are legal and that only a moderator/admin can approve/reject). `fetchListings` gained an `ownerId` filter so `owner=me` returns every status for "my listings" instead of the published-only default. `PATCH /v1/listings/{listingId}` wired into `v1-listings`, added to the OpenAPI contract, types regenerated. 7 new unit tests (23 total in the file) plus 7 new RLS assertions (34 total) covering a moderator seeing/approving another user's pending listing and a buyer's attempt having no effect.
- **Dashboard**: a real **Create Listing** page (`/listings/new`) that reuses the Schema Builder's `DynamicFormPreview` renderer to produce an actual submittable form from any category's live schema — not a mock-up, a second proof of "schema-driven, not screen-driven" after the Schema Builder preview itself. A **Listings** page (`/listings`) showing "My listings" (with role-appropriate transition buttons: submit for review, withdraw, archive, mark sold) and, for moderators/admins, a **moderation queue** with Approve/Reject.
- Verified the complete loop live in a browser: created an iPhone 15 listing (Electronics › Phones, a different vertical than every prior demo) as a seller, confirmed the Model dropdown correctly narrowed to iPhone options after selecting Apple, submitted for review, confirmed a self-approval attempt is rejected (403), switched to a moderator identity, saw it in the queue, approved it, and confirmed it became publicly visible — each step cross-checked against the real API, not just the UI.
- **Two real bugs caught and fixed along the way**: (1) the local dev gateway's router didn't forward `/v1/listings/{id}` (exact-path match only) — PATCH requests 404'd until fixed. (2) `useApi`'s effect dependency array used `[path, reloadKey, ...deps]`, a spread whose length varies by call site; collapsed to a single `JSON.stringify(deps)` key so the array length is always fixed — the safe pattern for a shared hook whose callers pass differently-shaped extra dependencies.

### Added — Phase 1 + Phase 2: Backend core + Dynamic Category & Attribute Engine ⭐, and the Dashboard (complete)

Reprioritized at the user's direction to push the configuration engine, category/attribute engine, listing engine, and dashboard ahead of iOS and further tooling — these are the platform's core business value. iOS (Phase 3/4) is untouched but unblocked.

**Local environment:** installed PostgreSQL 17 and Deno via Homebrew (both reversible CLI tools) to get a real database and runtime, since Docker (needed for `supabase start`/`functions serve`) isn't available here — see `backend/README.md` and `docs/ROADMAP.md`.

**Database** (`backend/supabase/migrations/`, all applied and verified against real Postgres):
- `auth`-stub schema (roles, `auth.uid()`/`role()`/`jwt()`) mirroring real Supabase for local RLS testing.
- `platform` (tenant, immutable audit_log), `identity` (profile + role/tenant assignment, custom-access-token-hook groundwork).
- `config` (versioned config/theme bundles, publish-by-insert-new-version-and-flip-active).
- `catalog` ⭐ — category tree (self-referencing, i18n, schema-version triggers), attribute groups/attributes/options/dependencies. The Dynamic Category & Attribute Engine's definition side.
- `listing` ⭐ — the hybrid attribute value store (ADR-0003): typed `listing_attribute_value` columns + a trigger-synced `attributes_index` JSONB projection, a type/validation-enforcing trigger, GIN indexing.
- Golden-path seed: 11 top-level categories, 14 subcategories, 44 attributes, 54 options, 2 dependencies — Cars/Apartments/Phones fully modeled, 8 others lightly modeled, proving the engine generalizes.
- A from-scratch SQL test harness (`backend/tests/sql/`, run via `backend/scripts/run-rls-tests.sh`) — 27 real assertions covering privilege escalation (blocked by trigger, not RLS), tenant isolation, public/owner/moderator listing visibility, attribute type enforcement, and dependent-option filtering.
- Two real security bugs caught and fixed before they shipped: missing schema/table `GRANT`s (RLS's `BYPASSRLS` doesn't imply object privileges) and a privilege-escalation path where `UPDATE ... WHERE id = auth.uid()` let a user change their own `app_role`/`tenant_id` (RLS restricts rows, not columns — closed with a trigger).

**Backend / Edge Functions** (`backend/src/` portable services + `backend/supabase/functions/`):
- `v1-config`, `v1-theme` — GET (ETag/304) + PATCH (publish, RLS-gated to admins).
- `v1-catalog` ⭐ — `GET /v1/categories/tree`, `GET /v1/categories/{id}/schema` (the composed contract DynamicForms/the dashboard render from), `GET/POST /v1/attributes/{id}/options`, plus the Schema Builder's write path (categories, attribute groups, attributes, options, dependencies).
- `v1-listings` ⭐ — `POST /v1/listings` (request-level validation against the category's schema, field-level errors, DB trigger as authoritative backstop) and `GET /v1/listings` (JSONB attribute filtering, cursor pagination).
- `v1-dev-auth` — local-only JWT minter standing in for GoTrue, deliberately excluded from `contract/openapi`.
- `_shared/db.ts` reproduces PostgREST's per-request `SET ROLE`/`SET request.jwt.claims` behavior directly against Postgres, so RLS applies faithfully without Docker; maps raw Postgres errors (RLS/FK/unique violations) to clean 4xx envelopes instead of leaking 500s.
- 16 Deno unit tests (mocked `QueryExecutor`) plus full live end-to-end verification (real HTTP → real Postgres) for every endpoint, run via `backend/scripts/serve-local.ts`.
- OpenAPI contract extended with the catalog/listings endpoint groups and schemas; `contract-types` regenerated and typechecked.

**Dashboard** (`dashboard/` — new Next.js 16 / App Router / TypeScript / Tailwind app, added to the npm workspace):
- **Schema Builder** ⭐ — category tree browser, per-category attribute editor (groups/attributes/options), and a **live dynamic-form preview** that renders the same composed schema contract the app would — proven live in a browser to correctly filter Model options by the selected Brand.
- **Config Studio** — locale/currency/module/support editing with publish, verified to persist.
- **Theme Studio** — semantic color-token editing with a live-updating component preview (button/card/badges), verified to persist and actually re-skin the preview.
- A dev-only identity switcher (mints tokens via `v1-dev-auth`) stands in for real login until Phase 1's GoTrue-backed auth flows exist.
- All three screens exercised directly in a real browser via the Preview tool (not just unit-tested): category/attribute/option creation, config publish, theme publish — each confirmed by re-querying Postgres afterward, not just trusting the UI.

### Added — Phase 0: Foundation & scaffolding (complete)
- Monorepo directory layout (`contract/`, `configs/`, `ios/`, `backend/`, `dashboard/`, `packages/`, `tooling/`), with a README in every scaffolded-but-not-yet-implemented directory stating its purpose and which phase populates it.
- OpenAPI v1 contract skeleton (`contract/openapi/v1/openapi.yaml`: health, config, theme endpoints, standard error/pagination envelopes) and Development Schema JSON Schema (`contract/schema/`: common/app/config/theme), plus `contract/examples` fixtures.
- Default reference config (`configs/clients/default`, fully populated) and `client_a` as a genuine partial-overlay stub (different locale default, currency, country, theme colors, module composition) proving the merge model; environment overlays for development/staging/production.
- `packages/config-validation` — Ajv2020-based validator: full-schema checks for `default` and every effective merged `(client × env)` config, structural (required-stripped) checks for overlay files. Caught and fixed a real bug (the `slug` pattern didn't allow `client_a`'s snake_case).
- `packages/contract-types` — generates TypeScript types from the contract (`json-schema-to-typescript` + `openapi-typescript`). Caught and fixed a real schema bug under `tsc --strict` (mixing named optional properties with typed `additionalProperties` on `modules`).
- CI pipeline (`.github/workflows/ci.yml`): config validation, contract-types generate+typecheck+drift-check, boundary lints, unit tests.
- Executable architecture-boundary lints (`tooling/lint/boundaries.sh`): no `import Supabase` outside `Networking`, no raw color/font literals outside `DesignSystem`, no Feature→Feature imports. Verified to actually catch injected violations, not just pass trivially.
- Local Supabase dev harness (`supabase init` run in `backend/`, `config.toml` customized: project id, seed path, anonymous sign-ins for guest browsing). Running the stack requires Docker, not installed in this session's environment — see `backend/README.md`.
- Refined [ADR-0004](docs/adr/0004-configuration-engine.md) with the exact per-file merge model (`app.json` never merges; `config.json`/`theme.json` deep-merge) discovered while implementing the config-validation package.
